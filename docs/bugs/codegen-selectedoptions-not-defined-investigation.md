# Investigation: Codegen GraphQL Document Validation fails on $selectedOptions

**Bug Slug:** codegen-selectedoptions-not-defined
**Investigation Date:** 2026-05-18
**Investigated by:** Claude (General Agent)

---

## Root Cause

The `PRODUCT_QUERY` operation in `app/routes/($locale).products.$productHandle.jsx` (lines 499–535) uses the variable `$selectedOptions` in the `PRODUCT_FRAGMENT` (line 482: `variantBySelectedOptions(selectedOptions: $selectedOptions)`) but does not declare `$selectedOptions` in the operation's variable signature (lines 500–504). GraphQL schema validation requires all variables used in an operation to be explicitly declared in the operation header.

---

## Mechanism

1. **The operation header** (lines 500–504) declares only three variables:

   ```graphql
   query Product(
     $country: CountryCode
     $language: LanguageCode
     $handle: String!
   )
   ```

2. **The fragment uses an undeclared variable** (line 482, inside `PRODUCT_FRAGMENT`):

   ```graphql
   selectedVariant: variantBySelectedOptions(selectedOptions: $selectedOptions) {
     ...ProductVariant
   }
   ```

3. **The fragment is included via spread** (line 506):

   ```graphql
   ...Product
   ```

   When the fragment is expanded into the operation, the `$selectedOptions` variable reference becomes part of the operation's body without a corresponding declaration in the operation header.

4. **GraphQL validation fails** because the Storefront API schema validates the entire operation (including expanded fragments) against declared variables. The variable `$selectedOptions` is used but not declared, violating GraphQL's `NoUnusedVariablesRule` and `KnownVariableNamesRule`.

5. **The type definition is correct** (`storefrontapi.generated.d.ts`, line 825–832) because the codegen tool scans the schema and correctly infers that `ProductQuery` requires `selectedOptions: SelectedOptionInput | Array<SelectedOptionInput>`. However, the codegen validator rejects the query string before types are generated.

---

## Suggested Fix Approach

Add `$selectedOptions: [SelectedOptionInput!]!` to the operation variable declaration in `PRODUCT_QUERY` (line 500), between the existing variables and before the closing parenthesis. The variable type should match the GraphQL schema definition for `variantBySelectedOptions(selectedOptions:)`, which accepts an array of `SelectedOptionInput` objects (non-nullable).

The operation header should become:

```graphql
query Product(
  $country: CountryCode
  $language: LanguageCode
  $handle: String!
  $selectedOptions: [SelectedOptionInput!]!
)
```

This will make the operation's variable declaration match the actual usage in the fragment.

## Regression Risk Areas

1. **The `PRODUCT_QUERY` operation itself** (`app/routes/($locale).products.$productHandle.jsx`, lines 499–535):

   - This is the primary site of the fix. Changes to the operation signature will affect how the loader calls `context.storefront.query()` at line 62.
   - The variables object passed at lines 63–68 already includes `selectedOptions` (sourced from `getSelectedProductOptions(request)` at line 59), so the fix is compatible with the existing code flow.

2. **TypeScript declarations** (`storefrontapi.generated.d.ts`, line 825–832):

   - These will be automatically regenerated by `npm run build` (which includes the `--codegen` flag). The generated `ProductQueryVariables` type already expects `selectedOptions`, so no manual type work is needed.
   - The build process will validate that the fixed GraphQL operation matches the schema.

3. **Similar pattern usage** — Search results show `selectedOptions` is used in other places:

   - `app/components/Cart.jsx`: references `merchandise?.selectedOptions` (reading from a loaded variant, not a query parameter — not affected).
   - `app/lib/seo.server.js`: iterates over `variant.selectedOptions` (post-fetch data, not a query variable — not affected).
   - `app/data/fragments.js`: `PRODUCT_CARD_FRAGMENT` requests `selectedOptions` as a field (not a variable — not affected).
   - Only the product page route uses `selectedOptions` as a query variable to select a specific variant.

4. **Related variant selection logic** (`app/routes/($locale).products.$productHandle.jsx`, lines 59, 82–84):

   - Line 59 calls `getSelectedProductOptions(request)` to extract selected variant options from the request URL.
   - Line 82–84 uses the returned `product` data (which includes `selectedVariant` resolved via the fixed query).
   - No changes needed here; the loader already passes `selectedOptions` to the query.

5. **No other routes use `variantBySelectedOptions`** (confirmed by grep):
   - Only the product page route (`($locale).products.$productHandle.jsx`) references this field.
   - Other product queries use `variants(first: N)` to fetch all variants and filter in memory (not affected).
